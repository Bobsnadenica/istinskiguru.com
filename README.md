# IstinskiGuru.com

IstinskiGuru.com is a static Bulgarian editorial and satirical website that catalogues public-facing online gurus, coaches, mentors, and course sellers. The project combines hand-written profile copy, public links, media assets, and generated share pages optimized for social previews.

The production site is published as static output on GitHub Pages at [www.istinskiguru.com](https://www.istinskiguru.com/).

## What This Repository Contains

- A static website built with plain HTML, CSS, and vanilla JavaScript
- Source profile content stored in `assets/`
- A Node-based content build pipeline
- Generated static profile pages in `profiles/`
- Generated optimized media in `site-assets/`
- SEO files such as `sitemap.xml`, `robots.txt`, `CNAME`, and `.nojekyll`

This repository intentionally keeps generated output checked in because GitHub Pages serves files directly and does not run the build step for you.

## Tech Stack

- HTML / CSS / vanilla JavaScript
- Node.js for the local build pipeline and preview server
- macOS `sips` for image optimization
- Swift + AppKit for Open Graph / social share card rendering
- GitHub Pages for static hosting

## Important Build Note

The full asset build currently assumes a macOS environment.

`npm run build:data` uses:

- `sips` to optimize and normalize images
- `swift` with `scripts/render-share-card.swift` to generate social share thumbnails

If you only edit layout, copy, or front-end behavior, you may not need to rebuild generated assets immediately. If you add or change profile source content in `assets/`, you should rebuild before publishing.

Edit these files directly:

- `assets/**`

- `supportus.html`
- `styles.css`
- `gallery.css`
- `gallery.js`

- `page-effects.js`
- `share-actions.js`
- `site-intro.js`
- `video-viewer.js`

Do not hand-edit these generated files unless you are debugging the build itself:

- `site-data.js`
- `site-assets/**`
- `profiles/**`

- `robots.txt`

They are regenerated by `npm run build:data` and can be overwritten at any time.

## Local Development

Install requirements:

- Node.js
- macOS if you need the full generated asset pipeline

Build generated content:

```bash
npm run build:data
```

Start the local preview server:

```bash
npm start
```

By default the site is served at:

```text
http://127.0.0.1:3000
```

The local server also exposes:

```text
/api/profiles
```

That API is for local preview and debugging only. Production on GitHub Pages is static and uses `site-data.js`.

## Publishing Workflow

Typical content workflow:

1. Add or update source content in `assets/`
2. Run `npm run build:data`
3. Review the generated changes in:
   - `site-data.js`
   - `site-assets/`
   - `profiles/`
   - `sitemap.xml`
   - `robots.txt`
4. Commit both source and generated output
5. Push to the branch that GitHub Pages publishes from

Because the site is static, GitHub Pages will only publish what is already committed.

## Social Sharing

Each profile gets its own static share page under:

```text
profiles/<slug>/index.html
```

Those pages contain:

- dedicated Open Graph metadata
- a generated share thumbnail
- a canonical URL
- structured data

This makes profile sharing work more reliably on platforms such as Facebook and LinkedIn than sharing only a homepage hash link.

Instagram and TikTok sharing are best-effort integrations that rely on device support for native sharing and file sharing.

## Public Repository Guidelines

This is a public repository, so please keep changes appropriate for a public codebase:

- Do not commit secrets, tokens, or private credentials
- Do not add non-public personal information
- Prefer public URLs and publicly visible source material only
- Keep documentation descriptive and professional
- Be careful with legal or factual claims in content copy

## Notes

- `CNAME` and `.nojekyll` are expected and should remain in place for GitHub Pages
- Generated profile pages are rebuilt from source content, so manual edits there will not last
- The site currently favors a static deployment model over a framework or CMS
